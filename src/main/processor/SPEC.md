# Event Processor Specification

## 1. Overview
The Event Processor is a self-contained module responsible for ingesting raw activity events (primarily screenshots) and transforming them into structured, queryable data for the user's personal context vault.

## 2. Core Responsibilities
- **Ingestion:** Listen for capture events from the main capture module.
- **Transformation:** Convert raw event data into semantic text and vector embeddings.
- **Storage:** Persist the transformed data into a local vector database (LanceDB).
- **Cleanup:** Ensure raw artifacts (screenshots) are deleted immediately after processing to respect user privacy and storage constraints.

## 3. Requirements (v1)
- **Encapsulation:** The processor must expose a clean API and not leak internal logic (OCR implementation, DB details) to the main process.
- **Method:** Use native macOS OCR (via Swift sidecar)
- **Storage Engine:** Use LanceDB for local vector storage.
- **Search Strategy:** Hybrid Search (Vector + FTS) for optimal retrieval.
- **Data Lifecycle:**
  1. Receive screenshot path.
  2. Extract text via OCR.
  3. Store text + metadata + embedding in DB.
  4. Delete original screenshot file from disk.

## 4. Data Model

### Input (Event)
What the processor receives from the Capture module:
```typescript
interface CaptureEvent {
  id: string;           // UUID
  filepath: string;     // Path to temporary PNG file
  timestamp: number;    // Unix timestamp
  displayId: number;    // Source display
}
```

### Storage (LanceDB Schema)
What gets persisted in the `events` table:
```typescript
interface StoredEvent {
  id: string;           // UUID (matches input)
  timestamp: number;    // Unix timestamp
  text: string;         // Raw text extracted via OCR (Must support FTS index)
  vector: number[];     // Embedding vector generated by Embedding model
}
```

## 5. Architecture & Flow
1. **Capture:** `main.ts` triggers `capture.ts`.
2. **Event:** `capture.ts` emits/callbacks with a `Screenshot` object.
3. **Process:** `Processor` receives the object.
   - Spawns/calls `ocr.swift` with the filepath.
   - Receives text output.
4. **Persist:** `Processor` writes the record to LanceDB table `context_events`.
5. **Cleanup:** `Processor` deletes the file at `filepath`.

## 6. Future Considerations
- **Enrichment + Cross-Platform:** Use LLMs for scene description beyond text, for Cross-Platform support and for aggregation of multiple events.
